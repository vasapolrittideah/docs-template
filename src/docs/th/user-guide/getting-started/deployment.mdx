# การ Deploy

docs-template สามารถ Deploy ไปยัง Platform ใดก็ตามที่รองรับ Next.js หน้านี้ครอบคลุม Deploy Target ที่พบบ่อยที่สุดสามแบบ: Vercel, Netlify และ Self-hosted Node.js Server

---

## ก่อน Deploy

ตรวจสอบให้แน่ใจว่าสิ่งต่อไปนี้พร้อมแล้วก่อน Deploy:

1. [Environment Variable](./authentication#environment-variables) ที่จำเป็นทั้งหมดถูกตั้งค่าใน Deployment Platform ของคุณ
2. OAuth Redirect URI สำหรับ Azure AD และ Google ถูกอัปเดตให้ชี้ไปยัง Production Domain ของคุณ

<Callout type="warning">
อัปเดต Redirect URI

หลัง Deploy ให้เพิ่ม `https://your-domain.com/api/auth/callback/azure-ad` และ `https://your-domain.com/api/auth/callback/google` เป็น Authorized Redirect URI ในการตั้งค่า OAuth Provider ของคุณ Authentication จะล้มเหลวหากไม่ทำขั้นตอนนี้
</Callout>

---

## Vercel

Vercel เป็น Deployment Platform ที่แนะนำสำหรับโปรเจกต์ Next.js

1. Push Repository ของคุณไปยัง GitHub, GitLab หรือ Bitbucket
2. ไปที่ [vercel.com](https://vercel.com/) และคลิก **Add New Project**
3. Import Repository
4. เพิ่ม Environment Variable ที่จำเป็นทั้งหมดใน **Environment Variables** ในการตั้งค่าโปรเจกต์
5. คลิก **Deploy**

Vercel จัดการ Build, Preview และ Zero-downtime Deployment โดยอัตโนมัติ

---

## Netlify

1. Push Repository ของคุณไปยัง Git Provider
2. ไปที่ [netlify.com](https://netlify.com/) และคลิก **Add new site** → **Import an existing project**
3. Import Repository
4. ตั้ง **Build command** เป็น `bun run generate-git-metadata && bun run build` และ **Publish directory** เป็น `.next`
5. ติดตั้ง [Next.js on Netlify](https://docs.netlify.com/integrations/frameworks/next-js/overview/) Build Plugin หากไม่ได้เพิ่มโดยอัตโนมัติ
6. เพิ่ม Environment Variable ที่จำเป็นทั้งหมดใน **Site configuration** → **Environment variables**
7. คลิก **Deploy site**

## การ Release อัตโนมัติ (CI)

Template นี้มี GitHub Actions Workflow (`.github/workflows/release.yml`) ที่รัน [semantic-release](https://semantic-release.gitbook.io/) ทุกครั้งที่มีการ Push ไปยัง `main` โดยจะเพิ่ม Version, สร้าง Changelog และสร้าง GitHub Release โดยอัตโนมัติ

### GH_TOKEN

Workflow นี้ต้องการ Repository Secret ชื่อ `GH_TOKEN` ซึ่งเป็น Personal Access Token (PAT) ที่ให้สิทธิ์ `semantic-release` ในการ:

- สร้างและอัปเดต GitHub Release
- Push Release Commit กลับไปยัง Repo (อัปเดต `CHANGELOG.md`, `package.json`, `bun.lock`)

เก็บเป็น Secret แทนการใช้ `GITHUB_TOKEN` เริ่มต้น เนื่องจาก `semantic-release` ต้องการ Push Commit ที่สามารถ Trigger Workflow อื่นได้ — Token ที่ Built-in มาไม่สามารถทำได้โดยตั้งใจ

#### การสร้าง Token

**Fine-grained token (แนะนำ):**

1. ไปที่ **GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens**
2. คลิก **Generate new token**, ตั้งค่า Expiration และ Resource Owner
3. ใน **Repository permissions** ให้สิทธิ์:
   - **Contents**: Read and write
   - **Issues**: Read and write
   - **Pull requests**: Read and write
   - **Metadata**: Read-only (เลือกอัตโนมัติ)
4. คลิก **Generate token** และคัดลอก

**Classic token:**

เลือก Scope `repo` — ครอบคลุมสิทธิ์ทั้งหมดที่ `semantic-release` ต้องการ

#### การเพิ่ม Secret

1. ไปที่ Repo ของคุณ → **Settings → Secrets and variables → Actions → New repository secret**
2. **Name:** `GH_TOKEN`
3. **Value:** วาง Token ที่สร้างไว้

<Callout type="info">
ใช้ Template นี้อยู่?

Secret จะไม่ถูกคัดลอกเมื่อสร้าง Repository ใหม่จาก Template แต่ละ Repository ต้องเพิ่ม `GH_TOKEN` Secret ของตัวเองเพื่อให้ Release Workflow ทำงานได้
</Callout>
